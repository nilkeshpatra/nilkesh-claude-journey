Payments Observatory — architecture requirements summary for solution architect review

ARCHITECTURE REQUIREMENTS

Payments Observatory

Solution design input

Problem statement

Customer-facing transaction data is currently fragmented across multiple downstream systems with no aggregation layer. There is no unified event stream, no cross-system correlation, and no real-time health signal. Issue detection is reactive and manual. The Observatory must resolve this with a purpose-built observability platform layered on top of existing payment infrastructure — without replacing it.

Layer 1

Source systems

Existing payment infrastructure (read-only)

Payment gateway · Fraud & risk engine · Ledger / core banking · Notification service · Customer identity store

Functional requirements

Data ingestion

  • Real-time event streaming from all source systems
  • Batch fallback for systems without event APIs
  • Canonical event schema across all sources
  • End-to-end transaction ID correlation

Processing

  • Sub-second latency for critical failure events
  • Stateful transaction lifecycle tracking
  • Configurable alerting rules engine
  • Anomaly detection on failure rates & latency

Storage

  • Hot store for live & recent transactions
  • Cold store for historical audit & trend analysis
  • Retention policy aligned to compliance obligations
  • Immutable audit log of all events

Presentation

  • Role-based views (exec / ops / engineer)
  • Transaction search & drill-down
  • Alert management & escalation workflow
  • Exportable incident reports

Non-functional requirements

Attribute Requirement
Availability 99.9% uptime — observatory must not become a payment dependency
Latency Critical alerts delivered within 30 seconds of event occurrence
Scalability Handle peak transaction volumes with headroom; horizontally scalable ingestion
Security Read-only access to source systems; data masking on PII; RBAC on all views
Resilience No single point of failure; graceful degradation if a source system is unavailable
Compliance Audit log retention aligned to PCI-DSS and applicable data residency rules

Key architecture decisions to resolve

Integration pattern

Event-driven vs polling per source system — depends on existing API capabilities

Build vs buy

Evaluate observability platforms (e.g. Datadog, Grafana) vs custom pipeline build

Data residency

Confirm cloud region constraints for payment event storage and processing

PII handling

Define masking / tokenisation strategy before ingestion design is finalised
